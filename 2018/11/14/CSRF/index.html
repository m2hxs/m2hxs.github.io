前言

​ 很多地方是借鉴大佬们的博客，此处仅为学习所用，请勿深究，感谢！

​ CSRF，全称Cross-site request forgery，翻译过来就是跨站请求伪造，是指利用受害者尚未失效的身份认证信息（cookie、会话等），诱骗其点击恶意链接或者访问包含攻击代码的页面，在受害人不知情的情况下，以受害者的身份向（身份认证信息所对应的）服务器发送请求，从而完成非法操作（如转账、改密等）。CSRF与XSS最大的区别就在于，CSRF并没有盗取cookie而是直接利用。在2013年发布的新版OWASP Top 10中，CSRF排名第8。

先简单的过一下基础的CSRF

1.简单的CSRF

​ 这种几乎是没有任何CSRF的防护措施，攻击者可以直接利用cookie登陆，达到利用别人的权限。

​ 这种连原始密码都不用输入（已登陆状态），这里示例代码就不贴了，直接用变量从前端接值，然后直接mysql_real_escape_string()函数处理，（转义 SQL 语句中使用的字符串中的特殊字符），传到数据库匹配。

​ 接下来，划重点！！

​ 这种修改密码的机制是直接提交form表单给数据库，数据库再执行insert函数。（可以通过生成html文件方式，来达到诱骗受害者的目的）

​ 同时，也可以通过url传递参数的形式，让数据库执行insert语句。

​ 稍进阶：构造一个新的html，里面通过img标签链接最开始生成的html文件，然后伪造一些前端页面。

2.中级别的CSRF

​

​ 这样的防护机制，很明显是检查了保留变量HTTP_REFERER（http包头的Referer参数的值，表示来源地址）中是否包含SERVER_NAME（http包头的Host参数，及要访问的主机名）。

​ 很简单，一句话：检查请求包里的REFERER，是否来源安全。

​ 怎么修改REFERER值这里就不说了。

3.高级的CSRF

本文主题来了：

​ 代码加入了Anti-CSRF token机制，用户每次访问改密页面时，服务器会返回一个随机的token，向服务器发起请求时，需要提交token参数，而服务器在收到请求时，会优先检查token，只有token正确，才会处理客户端的请求。

​ 流程：

A服务器中放置有js脚本，脚本里有iframe标签，跳转的请求服务器上。

​ 受害者访问并触发A服务器的js文件，跳转到请求服务器上。但是由于同源策略，A服务器中的js文件并不可以调用请求服务器上的资源。即js脚本无法接收（调用）到token值。

​ 与低级别不同的是，高级别需要用js脚本文件来调用（获取）受害者的token。而低级别则是直接加载页面，通过提交form表单上传来达到目的（直接验证cookie）。

​ 既然token都通不过，那自然无法提交表单。

同源策略及规避方法

Token

​ 存放于服务器与客户端的sessions中（session断掉，token消失），每次的token都是随机生成的。

Token的验证机制

​ 首先判断 session 中有没有 csrftoken，如果没有，则认为是第一次访问，session 是新建立的，这时生成一个新的 token，放于 session 之中，并继续执行请求。

​ 如果 session 中已经有 csrftoken，则说明用户已经与服务器之间建立了一个活跃的 session，这时要看这个请求中有没有同时附带这个 token，由于请求可能来自于常规的访问或是 XMLHttpRequest 异步访问（部分最新浏览器在主线程上的同步请求已经被弃用），我们分别尝试从请求中获取 csrftoken 参数以及从 HTTP 头中获取 csrftoken 自定义属性并与 session 中的值进行比较，只要有一个地方带有有效 token，就判定请求合法，可以继续执行，否则就转到错误页面。

​ 在客户端 html 中，主要是有两个地方需要加上 token，一个是表单 form，另一个就是URL。对于form，先遍历所有的 form，在 form 最后添加一隐藏字段，把 csrftoken 放入其中。然后，代码遍历所有的链接标记 a，在其 href 属性中加入 csrftoken 参数。

同步可以理解为单线程处理任务，异步可以理解为多线程处理任务

什么是Anti CSRF

  1. 用户访问某个表单页面。

  2. 服务端生成一个Token，放在用户的Session中，或者浏览器的Cookie中。 （生成 token 有很多种方法，任何的随机算法都可以使用，Java 的 UUID 类也是一个不错的选择。）

  3. 在页面表单附带上Token参数。

  4. 用户提交请求后， 服务端验证表单中的Token是否与用户Session（或Cookies）中的Token一致，一致为合法请求，不是则非法请求。

    ​ 这个Token的值必须是随机的，不可预测的。由于Token的存在，攻击者无法再构造一个带有合法Token的请求实施CSRF攻击。另外使用Token时应注意Token的保密性，尽量把敏感操作由GET改为POST，以form或AJAX形式提交，避免Token泄露。
